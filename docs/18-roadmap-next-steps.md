- 后端核心主链完成

## 3. 路线 A：收尾工程化
目标：把项目从“可交付骨架”推进到“更稳定可部署版”。

### 3.1 范围
1. 跑整套 Docker `build/up` 与容器联通性验证
2. 补齐 worker / beat 真正可运行闭环
3. 修复前端 lint warning
4. 优化前端包体与 chunk 拆分
5. 形成工程化收口文档与验证结果

### 3.2 交付标准
- `docker compose build` 通过
- `docker compose up` 后前后端/数据库/redis/worker/beat 基本联通
- 异步任务链至少具备最小真实消费能力
- 前端 warning 清零或明确保留说明
- 前端包体优化有实际结果

## 4. 路线 B：体验优化 / 真功能增强
目标：在工程化收尾后继续提升产品体验与真实能力。

### 4.1 范围
1. 更真实的 Celery 异步任务编排
2. 更好的 PDF / 多模态处理质量
3. 更细粒度知识点抽取与关联
4. 登录行为 / 观看时长 / 用户活动埋点增强
5. UI/交互打磨
6. 联调预览与产品级体验修补

### 4.2 交付标准
- 任务链路更接近生产可用
- 多模态输入质量提升
- 复习与知识加工结果更稳定
- 前端体验更完整、更顺手

## 5. 当前执行顺序
严格执行：
1. 先完成路线 A 文档与实施
2. 路线 A 验收后，再进入路线 B
