# 33-完整测试计划文档

## 1. 文档目的
本计划用于约束 `learning-system` 完整测试阶段的范围、层级、方法、进入/退出条件、脚本建设方向与结果判定标准。

该文档是 SA-15B / SA-15C / SA-15D 的直接上游依据，目标是让后续测试实现型 subagent 可以按文档直接开工，而不是再自行补测范围。

## 2. 测试目标
确保项目在当前可交付基线下满足以下目标：
- 主功能链路可用
- 关键 API 行为可用且具备基础错误路径覆盖
- viewer / admin 权限边界正确
- 前端关键页面可访问、可操作、可呈现
- Docker 部署链路可验证
- 路线 A / 路线 B 已完成能力不回退
- 已知低风险遗留项被明确标注，不与阻塞问题混淆

## 3. 测试范围

### 3.1 后端范围
覆盖以下模块：
- auth / health
- sources / notes / jobs
- review / FSRS / summaries / mindmaps
- settings / admin
- Celery app / worker / beat 基础可运行性
- Docker 下服务连通性与代理链路

### 3.2 前端范围
覆盖以下页面与行为：
- 登录页
- Notes：overview / generate / library / detail
- Review：overview / session / summaries / mindmaps
- Settings：ai / workspace / users / import-export / jobs
- 权限禁用态
- Markdown / HTML / Mermaid 渲染
- Jobs / activity 展示
- `watch_seconds` 上报行为

### 3.3 部署与集成范围
- `docker compose build`
- `docker compose up -d`
- frontend -> backend 代理
- postgres / redis / backend / worker / beat 联通性
- `.env.example` / `.env` 配置基线

## 4. 不在本轮直接展开的范围
以下内容可记录风险，但不是本轮完整测试阶段的扩张目标：
- 大规模性能压测
- 安全渗透测试
- 真机移动端适配专项
- OCR / 多模态模型效果质量评测
- Mermaid 包体进一步性能优化
- 已知 `watch_seconds` 触发访问次数重复累计问题的产品修复实现

## 5. 测试原则
- 先文档、后实现、再执行
- 先关键主链，后补充路径
- 先自动化可重复验证，后人工抽样补位
- 失败项、未覆盖项、阻塞项必须分开记录
- 不因为已知低风险遗留项阻断整轮测试，但必须在结果中标注影响范围
- 不直接修改业务代码来“配合测试通过”

## 6. 测试层级与实施要求

### 6.1 后端单元/服务测试
目标：验证核心服务逻辑与相对稳定的业务规则。

优先覆盖：
- JobService
- ReviewService
- SettingsAdminService
- NoteGenerationService
- 其他与当前主链直接相关、且已有测试基础或易于隔离的服务逻辑

要求：
- 以 `pytest` 为主
- 优先补服务层、schema 校验、状态转换与错误路径
- 避免写过度依赖外部模型能力的脆弱测试

### 6.2 API 集成测试
目标：验证 FastAPI 接口、鉴权、权限、输入输出契约与主流程闭环。

要求：
- 基于 `pytest + httpx/ASGI` 思路补齐
- 覆盖成功路径、未登录、权限不足、参数异常、资源不存在等关键错误路径
- 与 `docs/05-api-and-data-contracts.md` 保持一致

### 6.3 前端页面与交互测试
目标：验证页面加载、主交互、状态展示、渲染契约与权限禁用态。

要求：
- 先补测试脚手架，再落页面级与关键组件级测试
- 当前 `frontend/package.json` 尚无测试命令，因此 SA-15C 需先补前端测试运行基础
- 建议优先级：
  1. 页面渲染/路由可达
  2. 关键表单与交互
  3. API mock 下状态切换
  4. Mermaid / HTML / Markdown 渲染
  5. viewer/admin 禁用态

### 6.4 Docker / 部署验证
目标：验证项目以交付方式运行时的基础可用性。

要求：
- 至少覆盖 `docker compose config/build/up` 与健康检查
- 验证 frontend -> backend 代理链路
- 验证 postgres / redis / worker / beat 处于可观测或可判定状态

### 6.5 回归测试
目标：确保路线 A / B 已交付能力不回退。

要求：
- 结合前述自动化测试 + 部署验证做回归清单
- 回归必须覆盖：
  - 路线 A 工程化收口能力
  - 路线 B 产品增强能力
  - 权限禁用态
  - Jobs 可见性与 artifact 基础链路

## 7. 测试角色与交付边界

### 7.1 SA-15A（本文档阶段）
负责：
- 测试计划定稿
- 测试矩阵定稿
- 测试实施说明定稿

不负责：
- 业务代码修改
- 直接开测
- 替代 SA-15B / SA-15C / SA-15D 做实现或执行

### 7.2 SA-15B（后端测试增强与脚本）
负责：
- 后端 pytest 测试增强
- API / service 测试实现
- 生成可执行命令与基础说明

### 7.3 SA-15C（前端/集成测试脚本与验证工具）
负责：
- 前端测试基础设施搭建
- 页面/交互/渲染相关测试实现
- 必要的集成验证脚本或工具

### 7.4 SA-15D（完整测试执行与报告）
负责：
- 按既定顺序执行测试
- 汇总通过/失败/阻塞项
- 出具阶段测试报告与问题清单

## 8. 测试脚本建设方向

### 8.1 后端脚本方向
基于仓库现状：
- 已存在 `pyproject.toml`
- 已配置 `pytest`
- 已存在 `backend/tests`

因此 SA-15B 应采用：
- `pytest` 作为统一入口
- 按模块拆分测试文件
- 优先沉淀可重复执行的 fixture、鉴权辅助、测试数据构造工具

建议产出：
- `backend/tests/services/...`
- `backend/tests/api/...`
- 公共 fixture / 工具文件
- 必要时补测试说明文档

### 8.2 前端脚本方向
基于仓库现状：
- 存在 `frontend/package.json`
- 当前无测试脚本、无测试框架配置

因此 SA-15C 应先补：
- 前端单测/组件测试框架（例如 Vitest + Testing Library）或等价方案
- 对关键页面和关键组件建立稳定 mock 测试
- 对少量高价值链路补 e2e/集成脚本（若选择 Playwright，也应限定范围，不做无边界扩张）

建议优先建设：
- 路由级页面渲染测试
- 关键页面交互测试
- 渲染契约测试（Markdown / HTML / Mermaid）
- viewer/admin 权限禁用态测试

### 8.3 部署验证脚本方向
建议由 SA-15B / SA-15C 视实现情况补充最小验证脚本，SA-15D 统一执行：
- `docker compose config`
- `docker compose build`
- `docker compose up -d`
- 健康检查命令
- 代理可达性验证命令

进入 SA-15B / SA-15C 前必须满足：
- 测试计划与测试矩阵已定稿
- 测试范围无歧义
- 已知遗留项已在文档中注明

进入 SA-15D 前必须满足：
- 后端测试命令可执行
- 前端测试命令可执行

## 11. 退出条件
完整测试阶段可视为完成，至少需满足：
- 后端关键模块存在可执行自动化测试
- 前端关键页面/交互存在可执行测试或等价稳定验证脚本
- Docker/部署链路完成验证
- 形成通过项 / 未通过项 / 阻塞项 分类结果
- 形成阶段测试报告

## 12. 通过标准

### 12.1 文档阶段通过标准
- 范围、顺序、脚本方向、通过标准明确
- 后续 subagent 不需要再补定义即可开工

### 12.2 实现阶段通过标准
- 自动化测试脚本可执行
- 至少关键主链存在重复可运行用例
- 命令入口清晰
- 测试结果可复现

### 12.3 执行阶段通过标准
- 输出清晰的通过项/失败项/阻塞项
- 每类问题都有证据（命令、日志、截图或文本结果）
- 已知低风险遗留项不被误判为新阻塞缺陷

## 13. 缺陷分级建议
- 阻塞：无法启动、无法登录、关键主链完全不可用、部署链路中断
- 高：核心接口错误、权限越权、关键页面不可操作
- 中：局部功能异常、展示错误但有替代路径
- 低：文案、样式、统计口径偏差、非阻塞性能问题

其中当前已知低风险项包括：
- `watch_seconds` 上报会连带重复累计访问次数统计
- Mermaid chunk 体积仍偏大
- frontend 可能在 backend recreate 后短暂出现 upstream 陈旧问题
