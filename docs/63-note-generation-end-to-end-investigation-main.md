# 63-笔记生成全流程专项调查主任务文档

## 1. 背景
用户明确指出当前 AI 笔记生成流程理解仍不准确，要求按以下真实业务链路重新核对、测试与设计：
- docx / txt / md 等纯文本：文本内容 -> embedding 检索相关笔记 -> LLM 生成笔记
- PDF / 图片：OCR / 多模态识别转纯文本 -> embedding 检索相关笔记 -> LLM 生成笔记
- wav / mp3 等音频：语音识别转文本 -> embedding 检索相关笔记 -> LLM 生成笔记
- 每次笔记生成前都要用 embedding 模型检索相关笔记
- LLM 生成提示词与组织方式参考 `obsidian-spaced-recall`、`obsidian-study-notes`
- 需要向 LLM 注入当前日期/时间等基础上下文
- 笔记库按“学科分类”落库，由 AI 决定学科分类
- 每个笔记标题格式为：`标题-日期-时间`

## 2. 本轮目标
先做详尽调查、测试、信息收集、问题清单与解决方案设计，形成完整文档系统；在文档冻结前，不进入大范围实现。

## 3. 本轮交付物
- 需求拆解文档
- 当前仓库与现状分析文档
- 总体架构文档
- 模块边界文档
- API / 数据契约文档
- 实现总方案文档
- subagent 编排计划文档
- 测试与验证计划文档
- 配置与运行说明文档

## 4. 调查重点
1. 当前 provider 配置模型与 `ProviderType` 是否足够表达 embedding / llm / ocr / stt 四类能力
2. 当前 notes 生成链是否缺“embedding 检索相关笔记”阶段
3. 当前 prompt / 时间注入 / 学科分类 / 命名策略与用户要求差多少
4. 当前纯文本 / PDF / 图片 / 音频各自真实运行路径是什么
5. 真实 SiliconFlow 配置下，哪些链路能跑通、哪些结果质量有问题

## 5. 非目标
- 本轮先不直接承诺一口气完成所有重构
- 先不扩展无关 UI
- 先不做与本任务无关的部署改造
